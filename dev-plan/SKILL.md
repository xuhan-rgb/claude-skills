# Dev Plan

设计开发方案，识别关键决策点，给出推荐意见，等待用户批复后执行。

## 调用方式

```
/dev-plan <功能描述>                   # 分析需求、设计方案 → 生成 DESIGN_<项目名>.md (+.canvas)
/dev-plan 执行 DESIGN_xxx.md          # 按批复开始落地开发（推荐）
/dev-plan 开始 DESIGN_xxx.md          # 同“执行”
/dev-plan 部署 DESIGN_xxx.md          # 同“执行”（用于开始实施语义）
/dev-plan 修改 DESIGN_xxx.md          # 兼容旧命令（legacy）
/dev-plan 执行                         # 自动扫描 DESIGN_*.md 再定位
/dev-plan 修改                         # 兼容旧命令（legacy）
```

## 文件命名

每次设计生成**一对文件**：

```
DESIGN_<项目名>.md          ← 逻辑细节：决策点、实现步骤、文件清单
DESIGN_<项目名>.canvas      ← 可视化架构：模块关系、页面流转、数据流（可选）
```

| 命令                              | 生成文件                                            |
|-----------------------------------|-----------------------------------------------------|
| `/dev-plan UWB融合模块`           | `DESIGN_UWB融合模块.md`                             |
| `/dev-plan 用户认证系统`          | `DESIGN_用户认证系统.md` + `.canvas`                |
| `/dev-plan 管理后台`              | `DESIGN_管理后台.md` + `.canvas`                    |

### Canvas 生成规则

**自动生成 .canvas 的场景**（涉及可视化布局的项目）：
- 网页/前端开发（页面结构、组件关系、路由流转）
- 多模块系统（模块依赖、数据流向）
- 用户明确要求

**不生成 .canvas 的场景**：
- 单文件修改、小功能补丁
- 纯后端/算法类任务
- 用户明确不需要

## PLAN 模式（设计方案）

### 1. 需求分析

使用 Explore Agent 研究现有代码库：
- 相关模块的架构和接口
- 可复用的组件和模式
- 约束条件（依赖、平台、性能要求）

### 2. 设计方案

输出完整开发方案，包含：
- 整体架构
- 模块划分与文件结构
- 数据流与接口设计
- 实现步骤（分阶段）

### 2.5. 生成 Canvas 架构图（混合模式）

对符合条件的项目，额外生成 `DESIGN_<项目名>.canvas`（JSON Canvas 格式）。

采用**混合模式**：Group = 页面，Group 内节点 = UI 区块，Group 间连线 = 页面跳转。

#### Canvas 结构规则

**三层信息**：

| 层级       | Canvas 元素                | 表达内容                                       |
|------------|---------------------------|------------------------------------------------|
| 页面       | `group` 节点              | 每个页面/视图一个 group，label = 页面名        |
| UI 区块    | `text` 节点（在 group 内）| 页面内的功能区块，文字描述内容和行为            |
| 页面跳转   | `edge`（group 间连线）    | 页面间导航关系，label = 触发方式                |

**Group 内节点排列**：
- 节点**从上到下**排列，对应页面从上到下的视觉布局
- 同一行的节点**从左到右**排列，表示并排布局
- 每个节点的文字应简洁描述该区块的**内容和交互**，而非实现细节

**颜色约定**：

| 颜色         | 含义                                  |
|--------------|---------------------------------------|
| `"5"` 青色   | 主页面 Group（入口页、仪表盘）        |
| `"4"` 绿色   | 子页面 Group（功能页、详情页）        |
| `"3"` 黄色   | 决策点节点（对应 markdown 的 D1, D2） |
| `"1"` 红色   | 外部依赖/API 节点                     |
| `"6"` 紫色   | 共享组件 Group（导航栏、侧边栏等）    |
| 无色（默认） | 普通 UI 区块节点                      |

**节点文字格式**：
```
## 区块名称

功能描述：简述该区块做什么
交互：用户可以做什么操作
数据：展示什么数据、来源
```

**Canvas 的作用**：
- 用户可在 Obsidian 中**可视化编辑**：
  - 在 group 内**添加/删除节点** → 增减 UI 区块
  - **修改节点文字** → 改变区块的功能描述
  - **拖动节点位置** → 调整布局顺序（上下=垂直顺序，左右=并排）
  - **新增/删除 group** → 增减页面
  - **修改/删除连线** → 改变页面跳转关系
- 修改后的 canvas 作为**设计指导**，执行时 Claude 会读取并遵循

**Canvas 与 Markdown 的分工**：

| 内容           | Canvas (.canvas)                | Markdown (.md)               |
|----------------|--------------------------------|------------------------------|
| 页面结构       | Group + 内部节点（直观布局）    | 文字描述（详细）             |
| UI 区块        | 节点文字（做什么、怎么交互）    | 组件设计、props 定义         |
| 页面跳转       | Group 间连线 + label            | 路由配置                     |
| 决策点         | 黄色节点（标记位置）            | 完整方案对比 + 批复栏        |
| 实现细节       | ×                              | 代码逻辑、接口定义           |
| 文件清单       | ×                              | 完整表格                     |
| 共享组件       | 紫色 Group（独立于页面）        | 组件 API 说明                |

#### 前端专用模式：SVG 背景 + 控件覆盖（仅 dev-plan 使用）

仅在以下条件满足时启用该模式：
- 命令为 `/dev-plan ...`
- 任务是前端/Web 页面设计
- 用户希望在 Canvas 中拖动控件并驱动代码层同步修改

启用后结构规则：
- 页面预览用 `file` 节点（`.svg`）作为背景参考，不承载注释主信息。
- 可交互控件（Tab/按钮/入口）用 `text` 节点覆盖在 SVG 上，允许拖动。
- 控件节点连线到目标页面 `group`，`edge.label` 写触发动作（如 `点击 数据目录Tab`）。
- `nodes[]` 顺序保证：背景 SVG 在前（底层），控件节点在后（上层）。

控件节点建议模板：
```text
控件：数据目录 Tab
点击：切换到 /data-dirs
打开：数据目录页面
作用：进入目录复用管理
代码映射：AppLayout Tabs + App.tsx Route
```

### 3. 识别决策点

对方案中**存在多种可行路径**的环节，标记为决策点（D1, D2...），每个决策点包含：
- **推荐方案**：Claude 认为最优的选择 + 理由
- **备选方案**：其他可行路径 + 各自优劣
- **批复栏**：留空待用户决策

## FIX 模式（执行开发）

### 命令归一化（执行词）

以下命令词在 FIX 模式等价：
- `执行` = `开始` = `部署` = `修改(legacy)`
- Claude 统一按“开始落地开发”处理

### 0. 定位设计文档

与 project-audit 相同的定位逻辑：

```
/dev-plan 执行 DESIGN_xxx.md     → 直接打开该文件（推荐）
/dev-plan 开始 DESIGN_xxx.md     → 同上
/dev-plan 部署 DESIGN_xxx.md     → 同上
/dev-plan 修改 DESIGN_xxx.md     → 兼容旧命令
/dev-plan 执行                   → glob DESIGN_*.md → 0报错/1直接用/N让用户选
/dev-plan 修改                   → 兼容旧命令
```

### 0.5. 读取 Canvas 变更（如有）

打开 `.md` 文件后，检查同名 `.canvas` 是否存在：

```
DESIGN_管理后台.md      ← 读取决策点批复
DESIGN_管理后台.canvas  ← 读取架构变更（如有）
```

**Canvas 变更优先于 Markdown**（混合模式解析规则）：

| 用户操作                          | Claude 执行动作                                  |
|----------------------------------|--------------------------------------------------|
| Group 内新增 text 节点            | 在对应页面新增 UI 区块                            |
| 删除 Group 内的 text 节点         | 该 UI 区块不实现                                  |
| 修改节点文字                      | 按新描述实现该区块                                |
| 调整节点在 Group 内的上下位置     | 调整该区块在页面中的垂直顺序                      |
| 同一行放置多个节点（左右并排）    | 该行使用并排布局（flex-row / grid columns）       |
| 新增 Group                        | 新增一个页面/视图                                 |
| 删除 Group                        | 对应页面不实现                                    |
| 修改 Group 间连线                 | 按新的跳转关系实现路由/导航                        |
| 新增连线 label                    | label 文字作为导航入口的触发方式（如"点击头像"）   |
| 新增紫色 Group                    | 作为共享组件提取，被多个页面引用                    |

前端项目启用“SVG 背景 + 控件覆盖”时，额外解析规则：

| 用户操作                                      | Claude 执行动作                                                                 |
|-----------------------------------------------|---------------------------------------------------------------------------------|
| 拖动控件节点在页面中的左右位置                | 调整同级控件顺序与布局（如 Tab `items` 顺序、按钮左右位置、`flex`/`grid` 排布） |
| 拖动控件节点在页面中的上下位置                | 调整区块垂直顺序（组件渲染顺序、容器层级）                                       |
| 修改控件节点的“点击/打开/代码映射”文字        | 更新对应事件处理、路由跳转与目标组件接线                                         |
| 新增控件节点并连到目标页面 Group              | 新增可点击入口 + 新路由/弹窗打开逻辑                                            |
| 删除控件节点                                  | 移除入口控件及无用导航代码                                                       |
| 修改控件节点到页面 Group 的 edge 或 label     | 同步修改导航行为（`navigate` / 菜单跳转 / 弹窗触发）                            |

如果 `.canvas` 不存在，仅按 `.md` 批复执行。

### 1. 解析批复

| 批复内容               | 动作                            |
|------------------------|---------------------------------|
| 同意 / 确认            | 采用推荐方案                    |
| 备选方案名（如"方案B"）| 采用该备选方案                  |
| 自定义内容             | 按用户描述的方案执行            |
| 去掉 / 不做            | 跳过该环节                      |
| **空白**               | **采用推荐方案（默认信任）**    |

> **核心区别**：与 project-audit 不同，dev-plan 中**空白 = 采用推荐方案**。
> 因为 Claude 已给出推荐理由，用户未否决即视为默认同意。

### 2. 按阶段执行

按「实现步骤」中的阶段顺序开发：

1. **读取**决策点的最终方案（批复 > 推荐）
2. **创建/修改**文件，实现对应功能
3. 每个阶段完成后**汇报进度**，让用户确认再继续下一阶段
4. **更新** DESIGN 文档的步骤 checkbox

### 3. 阶段检查点

每完成一个阶段，暂停输出进度：

```
✅ 阶段 1 完成：基础框架

已创建/修改的文件：
  - src/xxx/new_module.h （新建）
  - src/xxx/new_module.cpp （新建）
  - CMakeLists.txt （修改：添加编译目标）

下一步：阶段 2 - 核心逻辑
继续？(y/n)
```

### 4. 更新设计文档

执行完成后更新 DESIGN 文档：
- 顶部状态改为 `✅ 已完成` 或 `🔄 进行中（阶段 X/Y）`
- 汇总表的决策点标注最终采用的方案
- 步骤 checkbox 打勾 `[x]`

## 与 project-audit 的对比

| 维度         | `/project-audit`          | `/dev-plan`                  |
|--------------|---------------------------|------------------------------|
| 用途         | 检查已有代码              | 规划新开发                   |
| 扫描对象     | 源代码、配置、git          | 需求、架构、决策             |
| 条目类型     | 风险/优化/工程规范         | 决策点（多选一）             |
| 空白批复     | **不修复**（保守）        | **采用推荐**（信任 Claude）  |
| 执行方式     | 一次性全部执行             | 分阶段执行，每阶段确认       |
| 生成文件     | `AUDIT_<方向>.md`         | `DESIGN_<项目名>.md`         |

---

## 注意事项

- 设计方案**不自动开始编码**，必须通过 `/dev-plan 执行`（或 `开始/部署`，兼容 `修改`）触发
- 执行过程按阶段推进，每阶段完成后等待确认
- 决策点的推荐方案基于对现有代码库的分析，优先复用已有模式
- 如果需求不清晰，先列出**待澄清问题**让用户补充，再生成完整方案
